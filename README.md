# 🚀 Hybrid CI/CD Pipeline: Jenkins (K3s) + Docker Host

هذا المستودع يحتوي على معمارية متقدمة وخفيفة (Lightweight Architecture) لعمليات التكامل والتسليم المستمر (CI/CD) مخصصة للسيرفرات السحابية الفردية (Single-Node Servers). 

يعتمد هذا الإعداد على تشغيل **Jenkins** داخل بيئة **K3s (Kubernetes)**، بينما تتم عمليات بناء الصور ونشرها (Build & Deploy) مباشرة على السيرفر المضيف (Host) باستخدام **Docker**، مع تخطي عوائق الشبكات الخاصة (مثل Tailscale/VPN) باستخدام خدمة **smee.io**.

---

## 🏗️ لماذا هذه المعمارية؟ (القرارات التصميمية)

تم تصميم هذه المعمارية لحل تحديات حقيقية متعلقة باستهلاك الموارد وأمان الشبكات في السيرفرات السحابية. إليك الأسباب التفصيلية وراء كل قرار:

### 1. لماذا يتواجد Jenkins داخل K3s؟
* **سهولة الإدارة:** الاستفادة من قوة K3s في إدارة الموارد، وقدرته على إعادة تشغيل Jenkins تلقائياً (Self-healing) في حال توقفه.
* **العزل (Isolation):** بقاء أدوات الإدارة معزولة في بيئة Kubernetes بعيداً عن ملفات وتطبيقات السيرفر المضيف.

### 2. لماذا نبني الصور (Docker Build) على السيرفر وليس داخل K3s؟
* **مشكلة K3s و Docker:** بيئة K3s تعتمد على `containerd` ولا تحتوي على محرك Docker الافتراضي (Docker Shim).
* **استهلاك الموارد:** محاولة بناء الصور داخل K3s (سواء عبر Kaniko أو Pods مؤقتة) تستهلك كمية هائلة من الذاكرة (RAM) والمعالج (CPU) في كل عملية بناء.
* **الحل (DooD عبر SSH):** بدلاً من ذلك، يقوم Jenkins بالاتصال بالسيرفر المضيف عبر SSH ويستخدم محرك Docker المثبت هناك. هذا يتيح لنا استخدام **كاش دوكر (Docker Cache)** مما يسرّع عملية البناء بشكل مضاعف ويوفر الموارد بشكل كبير.

### 3. لماذا نستخدم Smee.io؟
* السيرفر محمي بشبكة خاصة (VPN/Tailscale) وغير متاح مباشرة للإنترنت العام لزيادة الأمان.
* هذا يمنع GitHub من إرسال إشعارات التحديث (Webhooks) مباشرة إلى Jenkins.
* **الحل:** تعمل خدمة **[smee.io](https://smee.io/)** كقناة وسيطة (Proxy). يستقبل smee الطلبات من GitHub، ويقوم عميل smee المثبت على السيرفر بسحب هذه الطلبات وتمريرها داخلياً إلى Jenkins بشكل آمن دون الحاجة لفتح بورتات للسيرفر.

### 4. النشر عبر Docker Compose (مع إمكانية النشر على K3s)
* في هذا الـ Pipeline، يتم النشر (Deployment) باستخدام `docker-compose` على السيرفر المضيف لسهولته ومناسبته للتطبيقات الحالية.
* **ملاحظة للمطورين:** يمكنك بسهولة تعديل المرحلة الأخيرة (Deploy Live) في الـ `Jenkinsfile` لاستخدام أوامر `kubectl apply -f deployment.yaml` إذا كنت ترغب في نشر التطبيق كـ Pods داخل K3s بدلاً من Docker.

## 🛠️ متطلبات التشغيل (Prerequisites)

لتطبيق هذه المعمارية، يجب أن تمتلك:
1. سيرفر سحابي يحتوي على **Docker** مثبت على النظام المضيف.
2. بيئة **K3s** تعمل على نفس السيرفر وتحتوي على Jenkins (أو أي مكان آخر يمكنه الوصول للسيرفر عبر SSH).
3. قناة Webhook من **smee.io** متصلة بـ Jenkins.
4. حساب في **Docker Hub** أو أي Container Registry آخر.

---

## ⚙️ كيفية الاستخدام

1. **إضافة بيانات الاعتماد في Jenkins (Credentials):**
   * أضف `Username with password` لحساب Docker Hub الخاص بك وقم بتسمية الـ ID بـ `docker-hub-creds`.
   * أضف `SSH Username with private key` للسماح لجينكنز بالدخول للسيرفر المضيف وقم بتسمية الـ ID بـ `server-ssh-key`.

2. **تعديل متغيرات البيئة (Environment Variables):**
   * افتح ملف `Jenkinsfile` وقم بتعديل قسم `environment` ليناسب مشروعك (اسم الصورة، رابط الـ Git، الـ IP الخاص بالسيرفر، ومسارات المجلدات).

3. **إنشاء Pipeline:**
   * في Jenkins، قم بإنشاء Pipeline جديد واربطه بهذا المستودع ليقوم بقراءة ملف `Jenkinsfile` تلقائياً.
